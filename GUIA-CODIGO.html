1. Visión general del sistema

DEVOPS Qué es esto, en una línea: un SaaS multi-tenant para talleres mecánicos. Un backend Spring Boot sirve todo (API + frontend empaquetado) en una sola URL.
  • Puerto: 8586 (cambia en application.properties)
  • Perfil activo: dev → H2 en memoria (al arrancar, el seeder crea datos demo)
  • Credenciales seed: test@gmail.com / password (busca DatabaseSeeder.TEST_EMAIL)
  • Frontend: Angular 19 SPA, embebida en src/main/resources/static/ y servida por Spring Boot
  • Otro proyecto conviviendo: prjcomidalpaso (restaurantes) en :8080 — no se mezclan

3. Backend — arquitectura hexagonal

BACKEND Lo más importante del backend: tiene 11 bounded contexts (carpetas independientes dentro de com.utp.myapp), cada uno sigue el mismo molde de 4 capas.

3.1 Los 11 bounded contexts

Cada contexto vive en src/main/java/com/utp/myapp/<nombre>/. Crucial: los contextos no se importan entre sí directamente. La composición (un caso de uso que necesita leer del contexto reminder) se hace vía composición de beans en el contexto que llama, no con import.

ContextoResponsabilidadEndpoint raíz
authLogin, register, refresh, hashing, JWT/api/auth/*
tenantTalleres (tenant), su config/api/tenant/*
salesClientes y sus direcciones/api/customers
vehicleVehículos, imágenes, kilometraje/api/vehicles
workorderÓrdenes de trabajo y presupuestos/api/work-orders
reminderRecordatorios de mantenimiento/api/reminders
catalogRepuestos / categorías de partes/api/catalog/parts
analyticsStats para el dashboard/api/analytics/*

3.2 Las 4 capas (el molde hexagonal)

Cada contexto (excepto shared y publicapi) sigue este molde:

<contexto>/
├─ domain/
│  ├─ model/
│  │  ├─ aggregates/        # Customer, WorkOrder, Reminder, Tenant, Vehicle...
│  │  ├─ valueobjects/       # Address, Email, TenantId (records inmutables)
│  │  └─ exceptions/         # InvalidCredentialsException, etc.
│  └─ repository/            # Solo INTERFACES (puertos de salida)
│
├─ application/              # opcional — algunos contextos son pequeños y no lo usan
│  ├─ command/               # CreateCustomerCommand, UpdateCustomerCommand...
│  ├─ query/                 # GetCustomerByIdQuery, SearchCustomersQuery...
│  ├─ handler/               # CreateCustomerCommandHandler (un handler por command/query)
│  ├─ dto/                   # CustomerDto, CustomerSummaryDto (lo que viaja por HTTP)
│  └─ assembler/              # CustomerAssembler (convierte Domain ↔ DTO)
│
├─ domain OR interfaces/     # si el contexto es simple, los controllers viven aquí directo
│  └─ rest/                  # CustomerController, etc.
│
└─ infraestructure/
   ├─ persistence/jpa/
   │  ├─ entities/           # CustomerEntity (modelo JPA)
   │  ├─ mappers/             # CustomerMapper (Entity ↔ Aggregate)
   │  ├─ adapters/            # CustomerRepositoryAdapter (implementa ICustomerRepository)
   │  └─ repositories/        # JPACustomerRepository (interfaz Spring Data)
   └─ security/                # JwtTokenProvider, SecurityConfig, JwtAuthenticationFilter
La regla cardinal: las dependencias apuntan hacia adentro. El domain no sabe nada de Spring, JPA, ni HTTP. Si necesitas importar algo de org.springframework en un archivo de domain/, algo está mal.

Decisión de seguridad a notar: en SecurityConfig la regla !request.getRequestURI().startsWith("/api/") abre todas las rutas que NO empiezan con /api/. En teoría eso da acceso público al SPA — pero como el SPA solo expone /login, /register, y bajo /app/**, la auth real la hace el AuthGuard del frontend (no mira window en el JVM). 3.5 Cómo funciona la auth JWT

El flujo es clásico pero hay dos token types en juego:

  1. El usuario hace POST /api/auth/login con email+password.
  2. El LoginCommandHandler valida, genera dos tokens:
    • Access token (24h): lleva tenantId, role, userId. Es el que va en cada request.
    • Refresh token (7d): lleva "type": "REFRESH". Sirve solo para renovar.
  3. El JwtAuthenticationFilter extrae el header Authorization: Bearer ..., valida firma+expiración, y rellena el SecurityContext con el user.
  4. Si la ruta requiere auth y no hay token, devuelve 401.

Si necesitas añadir un nuevo endpoint protegido, solo anota con @PreAuthorize o déjalo bajo anyRequest().authenticated() (que es el default en SecurityConfig).

3.6 Multi-tenant — cómo se aísla la data

Real multi-tenancy por columna tenant_id. El aislamiento se hace en dos puntos:

  1. Capa de aplicación: el JwtTokenProvider mete el tenantId en el access token.
  2. Capa de JPA: el CustomerRepositoryAdapter (y similares) construyen queries con findByTenantId(tenantId, ...) — siempre filtran por tenant.
NO te confíes. Si añades un nuevo endpoint y usas findAll() en vez de filtrar por tenant, devuelves datos de TODOS los tenants. Cada adaptador tiene su propia findByTenantId(...) — verifica que la use.

La TenantContext (ThreadLocal) es inyectada por TenantFilter y consumida por TenantInterceptor cuando un handler lo pide. No abuses: la mayor parte del tiempo se pasa el tenantId explícito.

3.7 Patrón CQRS — los handlers

En cada comando/query hay una clase Command (record) + un Handler (@Service) + opcionalmente Assembler (DTO ↔ Domain):

// application/command/CreateCustomerCommand.java
@Getter @RequiredArgsConstructor @Builder
public class CreateCustomerCommand {
    private final String name;
    private final String lastName;
    private final String dni;
    private final String email;
    private final String phone;
    private final AddressDto address;
}

// application/handler/CreateCustomerCommandHandler.java
@Service
public class CreateCustomerCommandHandler {
    @Transactional
    public CustomerDto handle(CreateCustomerCommand cmd) {
        Customer c = assembler.toDomainCreate(cmd);
        c = customerRepo.insert(c);
        return assembler.toDto(c);
    }
}

El CustomerController solo arma el Command y llama al handler:

@PostMapping
public ResponseEntity<ApiResponse<CustomerDto>> create(
        @RequestBody CreateCustomerCommand command) {
    CustomerDto result = createHandler.handle(command);
    return ResponseEntity.status(HttpStatus.CREATED)
            .body(ApiResponse.created(result));
}

Si añades una nueva operación de cliente, creas UpdateXxxCommand.java + UpdateXxxCommandHandler.java. El controller solo crece en líneas de wiring.

4. Frontend Angular — estructura

4.1 Routing con lazy loading

src/app/app.routes.ts define 3 grupos de rutas:

  • '' → HomeComponent (landing, ruta pública)
  • login, register → auth screens (públicas)
  • app → MainLayoutComponent (con sidebar/header, requiere auth vía AuthGuard) que carga perezoso: dashboard, customers, vehicles, work-orders, reminders, analytics, catalog, settings

Cada componente se carga con:

loadComponent: () => import('./features/customers/customers.component').then(m => m.CustomersComponent)

Resultado en el bundle: el chunk de customers, vehicles, etc. se descarga cuando entras a esa ruta, no en la primera carga. Útil para móviles.

4.2 AuthInterceptor — la clave del transparente

Cada vez que un componente hace api.get('/customers'), el interceptor añade Authorization: Bearer <token> automáticamente. El componente nunca concatena headers:

intercept(req, next) {
  const token = this.authService.getToken();
  if (token) {
    const cloned = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` }
    });
    return next.handle(cloned);
  }
  return next.handle(req);
}

4.3 ApiService — la única forma de hablar con la API

La regla es: todos los requests pasan por api.service.ts, que hace map(res => res.data) automáticamente (recordá que el backend envuelve todo en {success, message, data, timestamp}):

get<T>(endpoint: string, params?: Record<string, any>): Observable<T> {
    return this.http.get<ApiResponse<T>>(`${environment.apiUrl}${endpoint}`, ...)
      .pipe(map(res => res.data));
}

Si en un componente ves this.http.get(...) directamente — estás haciendo mal. Usa siempre this.api.get(...) para que la desestructuración de la respuesta sea correcta.

4.4 Charts con ng2-charts v7 — el fix clave

Bug conocido: con ng2-charts 7+, si te olvidas de provideCharts(withDefaultRegisterables()) en app.config.ts, los charts se renderizan vacíos sin error visible. Los controllers (bar, doughnut) y scales (linear) no se registran con Chart.js.

src/app/app.config.ts debe tener:

import { provideCharts, withDefaultRegisterables } from 'ng2-charts';

providers: [
  ...,
  provideCharts(withDefaultRegisterables()),
  ...
]

Sin esa línea → pantallas en blanco con tamaño 595×288 (toman el espacio) pero 0 píxeles pintados. Síntoma idéntico a "el backend no devuelve datos" pero en realidad es la librería.

4.5 Cómo se construye y empaqueta

  1. pnpm run build en frontend/ genera dist/browser/ (~500KB de bundle + chunks lazy).
  2. El plugin maven-resources-plugin en pom.xml copia automáticamente frontend/dist/browser/* → src/main/resources/static/ en cada mvn package.
  3. Spring Boot sirve esos archivos en la raíz del JAR.
  4. El SpaForwardController forwarda las rutas SPA a index.html para que Angular Router tome el control.
Detalle: el plugin agrega encima pero no limpia archivos obsoletos. Si renombras main-XXX.js, el viejo queda en src/main/resources/static/ y se sube al JAR. Hay que limpiar manualmente con rm src/main/resources/static/main-OLD.js o usar mvn clean. Si no, el navegador cachea el viejo.

6. Trampas conocidas

bug Customer CRUD falla con city NOT NULL

Si rotas al pasado (commit anterior con @Column(nullable=false) en AddressEmbeddable), el POST de customers falla incluso si el address está vacío. Fix mantenido: las 4 columnas de AddressEmbeddable tienen @Column sin nullable, y el record Address en domain no lanza excepciones.

bug Refresh token devuelve success=false

Si rotas a un commit antes de implementar POST /api/auth/refresh, el endpoint no existe. El SecurityConfig lo lista en permitAll pero el controller no existía hasta que se añadió RefreshTokenCommandHandler.

bug Charts vacíos en dashboard

Si provideCharts(withDefaultRegisterables()) no está en app.config.ts, los charts se renderizan como canvas vacíos de tamaño 595×288 sin errores en consola. Síntoma sutil.

atención AddBundle sin limpiar viejos

Después de un pnpm run build, si el nombre del bundle cambia (p. ej. main-ZFS.js por uno nuevo), los archivos viejos quedan en src/main/resources/static/. mvn clean o borrado manual antes de mvn package.

atención IPv6 rota

Windows rota direcciones IPv6 cada cierto tiempo. Si la URL deja de funcionar, ejecuta curl https://api64.ipify.org y actualiza.

nota open-in-view=false

En application.properties está spring.jpa.open-in-view=false. Esto evita queries a la BD en el momento de serializar el response (un anti-patrón clásico de JPA). Implicación: si tu handler devuelve un agregado con relaciones lazy y luego en el DTO accedes a ellas, te va a tirar LazyInitializationException. La solución es cargar las relaciones en el @Transactional handler, no en el assembler.

nota ddl-auto=update

Hibernate crea y actualiza tablas automáticamente. En dev está bien. En prod debes usar migraciones (Flyway/Liquibase). Hoy por hoy cada arranque del backend es seguro contra pérdida de data — pero los índices no se optimizan.

truco CORS solo para localhost:4200

En SecurityConfig.corsConfigurationSource() solo está http://localhost:4200. Si abres el frontend desde otra IP, CORS bloqueará. Solución: usar el mismo origen del backend (como está configurado en prod) o agregar la IP a esa lista.

7. Glosario rápido

TérminoSignificado
Aggregate RootUna entidad con identidad propia (Customer, WorkOrder). Se carga y persiste completa, no por partes. Vive en domain/model/aggregates/.
Value ObjectDato sin identidad, inmutable. Address, Email, TenantId, Money. Son records Java.
Repository (interface)Puerto de salida del dominio. Define QUÉ se puede hacer con los agregados, no CÓMO. Vive en domain/model/repository/. Lo implementa el adaptador en infraestructure.
AdaptadorImplementación JPA del repositorio. Convierte Aggregate ↔ Entity ↔ fila SQL. Mapea datos pero nunca toca el dominio.
Handler (CQRS)@Service que ejecuta una intención (crear/actualizar/buscar). Un Command o Query → un Handler.
AssemblerConvierte Domain ↔ DTO. Los DTOs son los que viajan por HTTP.
PermitAll / authenticatedReglas en SecurityConfig. Cualquier ruta no listada explícitamente cae en anyRequest().authenticated().
Stub de seed (H2 dev)Usuario test@gmail.com / password, tenant test-tenant-001, con 3 clientes/vehículos y órdenes variadas. Solo existe si arrancas con perfil dev (que es el default).
